# 1 Введение

Данный документ описывает план тестирования приложения «Falling box». Он предназначен для людей, выполняющих тестирование приложения, в частности тестировщиков. Цель тестирования - проверка соответствия между реальным поведением программы и её ожидаемым поведением на основе [требований](https://github.com/OdareNNbI/Falling-box/blob/master/docs/SRS.md).

<a name="items"/>

# 2 Объект тестирования

На основе требований можно выделить следующие атрибуты качества: 
* Функциональность - выполнение все требований; 
* Удобство использования - качество обработки пользовательского ввода с игрой;
* Надежность - сохранение пользовательских данных;

<a name="risk"/>

# 3 Риски

1. В зависимости от версии Android, на более старых игра может работать некорректно.
2. В зависимости от разрешения экранов некоторые игровые объекты могуть быть видныне полностью.
3. Скорость обработки игровой логики зависит от процессора девайса.
4. Качество изображения зависит от максимального разрешения экрана.

<a name="features"/>

# 4 Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения.

Аспекты, подвергаемые тестированию:  
* вход в приложение;  
* переход между игровыми состояниями;  
* корректрое сохранение и отображение максимального счета;
* корректное отображение счета;
* покупка новой коробки;
* проигрыш;
* собирание звездочек;
* обработка нажатий пользователя.

##### Возможность входа в приложение
Этот вариант использования небходимо протестировать на:
* Вход без Интернет подключения;
* Вход с Интернет подключением.

##### Переход между игровыми состояниями
Этот вариант использования небходимо протестировать на:
* Все различные переходы между игровыми состояниями: из меню в игру и в магазин, из магазина в меню, после проигрыша в меню.

##### Корректрое сохранение и отображение максимального счета
Этот вариант использования небходимо протестировать на:
* Корректное отображение максимального счета в поп-апе проигрыша;
* Сохранение максимального количества очков при выхое из приложения в различных игровых состояний - из меню, самой игры, экрана проигрыша и магазина.

##### Корректное отображение счета
Этот вариант использования небходимо протестировать на:
* Корректное отображение счета после проигрыша;
* Корректное отображение счета при сворачивании и внось разворачивании игры.

##### Покупка новой коробки
Этот вариант использования небходимо протестировать на:
* Покупку коробки с количеством звездочек, меньшим чем стоимость коробки;
* Покупка коробки при достаточном кодичестве звездочек;
* Сохранение купленной коробки после перехода по состояниях;
* Сохранение купленной коробки после выхода из игры и последующего входа в неё.

##### Проигрыш
Этот вариант использования небходимо протестировать на:
* Правильное соприкосновение коробки с платформами;
* Правильное соприкосновение коробки с шипами;
* Правильное движение камеры;
* Проигрыш при падении за экран камеры.

##### Собирание звездочек
Этот вариант использования небходимо протестировать на:
* Правильное соприкосновение коробки с звездочками;
* Правильное отображение количества звездочек в магазине;
* Правильное отображение количества звездочек в игре;
* Правильное сохранение количества звездочек после покупки коробки.

##### Обработка нажатий пользователя
Этот вариант использования небходимо протестировать на:
* При нажатии по экрану коробка падает вниз.

##### Нефункциональные требования

* Производительность - отсутствие фризов во время использования приложения;
* Понятный гемплей - понимание того, что надо делать с 1-3 игры.

# 8 Выводы

Исходя из результатов тестирования можно сделать вывод, что приложение почти полностью соответствует функциональным требованиям. Также можно сделать вывод, что приложение не удовлетворяет некоторым нефункциональным требованиям. Функционал приложения является не большим, но позволяющий позволяет скороротать время в метро, автобусе и т.д. В будущем хотелось бы добавить больше различных коробок и, возможно, новый режим игры.
